## CAP-2: Agent Configuration Unification

### Requirement: Canonical Agent Config Location

Each client's agent configuration MUST be stored in a single JSON file at:

```
backend/app/agents/configs/{client_id}.json
```

The JSON file MUST conform to this schema (all fields required):

| Field | Type | Description |
|-------|------|-------------|
| `client_id` | string | Matches the filename stem (e.g. `"quintana-seguros"`) |
| `broker_name` | string | Display name of the insurance broker |
| `agent_name` | string | Display name of the AI agent |
| `elevenlabs_agent_id` | string | ElevenLabs agent ID for this client |
| `model` | string | LLM model identifier (e.g. `"gpt-4o"`) |
| `temperature` | number | LLM temperature (0.0–1.0) |
| `max_tokens` | integer | Max response tokens |
| `tools_enabled` | array of strings | Tool names available to this agent |
| `language` | string | BCP-47 language tag (e.g. `"es-AR"`) |

#### Scenario: Config file exists for active client

- GIVEN `backend/app/agents/configs/quintana-seguros.json` exists and is valid
- WHEN the webhook receives a request with `client_id = "quintana-seguros"`
- THEN the webhook MUST load agent parameters (model, temperature, max_tokens) from the file

#### Scenario: Config file not found — fallback

- GIVEN no config file exists at `backend/app/agents/configs/{client_id}.json`
- WHEN the webhook receives a request for that `client_id`
- THEN the webhook MUST return HTTP 404 with body `{"error": "agent config not found: {client_id}"}`
- AND MUST NOT fall back silently to hardcoded defaults

### Requirement: System Prompt Loaded from insurance_agent.py

The webhook MUST NOT contain an inline hardcoded system prompt.
The system prompt MUST be generated by calling `render_system_prompt(client, lead)` from `backend/app/prompts/insurance_agent.py`.

#### Scenario: System prompt rendered at runtime

- GIVEN a valid `client` record and an optional `lead` record
- WHEN the webhook builds the messages array for GPT-4o
- THEN the system message content MUST equal the output of `render_system_prompt(client, lead)`
- AND MUST NOT contain literal Python string concatenation building a prompt inline in `webhook.py`

#### Scenario: No lead available — graceful render

- GIVEN a valid `client` record and `lead = None`
- WHEN `render_system_prompt(client, None)` is called
- THEN it MUST return a fully rendered string with no `{placeholder}` tokens remaining

## CAP-3: WebSocket 1006 Close Handling

### Requirement: Distinguish Connection Loss from Normal Termination

The demo UI at `backend/app/static/index.html` MUST differentiate WebSocket close codes in `ws.onclose`.

| Close Code | Trigger | UI Message | UI State |
|------------|---------|------------|----------|
| 1000 | Normal close or `end_call` tool | "Conversación finalizada" | neutral (no dot color) |
| 1006 | Abnormal / network loss | "Se perdió la conexión" + reconnect button | error (red dot) |
| Any other | Unexpected | "Se perdió la conexión" + reconnect button | error (red dot) |

The reconnect button MUST re-invoke `startConversation()` and MUST be removed from the DOM once reconnection is attempted.

#### Scenario: Normal conversation end (code 1000)

- GIVEN a WebSocket connection is open
- WHEN it closes with code 1000
- THEN the status text MUST read "Conversación finalizada"
- AND no reconnect button is shown
- AND the dot color MUST NOT be red

#### Scenario: Unexpected disconnection (code 1006)

- GIVEN a WebSocket connection is open
- WHEN it closes with code 1006
- THEN the status text MUST read "Se perdió la conexión"
- AND a reconnect button MUST appear in the UI
- AND the dot MUST be in error (red) state

#### Scenario: Agent calls end_call tool

- GIVEN the agent sends an `end_call` tool invocation during a conversation
- WHEN the WebSocket subsequently closes
- THEN the UI MUST display "Conversación finalizada" (same as code 1000)
- AND no reconnect button is shown
